World-space UI contract

Render surfaces are UI hosts, not decorative canvases.

The API names below are directional, not frozen. The invariant is the important part: real retained children and layout, explicit graphics access, isolated render state, and input transformed back into the surface coordinate system.

Plane / render target / world quad

RenderSurface2D

Host a Grid, StackPanel, Canvas, templated controls, focus scopes, motion, and routed input inside a 2D scene region. Use the surface as a normal retained element around custom game rendering.

  • Nested controls participate in measure and arrange.
  • Pointer coordinates map through viewport, scale, camera, and surface transforms.
  • Surface content can be cached or redrawn on invalidation.
  • Render state is restored before Cerneala continues drawing.
Mesh / material / projected target

RenderSurface3D

Render a complete retained subtree into world space, then keep control semantics intact. A cockpit screen, terminal, map table, weapon display, or in-world editor remains a real UI tree instead of a texture with fake buttons.

  • Ray hits map into local layout coordinates.
  • Focus, capture, routed events, and accessibility remain root-owned.
  • Occlusion and camera policy stay explicit game decisions.
  • Render-target lifetime follows the surface and graphics device.
MODE A

Surface-specific event

The safe default. Subscribe to a render event and receive a scoped frame with the surface bounds, transforms, timing, graphics device, and an already prepared SpriteBatch.

surface.Render += (_, frame) =>
{
    frame.SpriteBatch.Draw(
        radarTexture,
        frame.Bounds,
        Color.White);
};
MODE B

Direct RAW access

The escape hatch. Take explicit responsibility for the graphics device, targets, effects, batches, depth, rasterizer and blend state. Cerneala fences the operation, then restores its own pipeline.

surface.RenderMode = RenderMode.Raw;
surface.RenderRaw += (_, raw) =>
{
    raw.GraphicsDevice.SetRenderTarget(target);
    DrawTheWildStuff(raw);
};

Illustrative API only. Final names wait for architecture, device-loss, state-isolation, nested-root, input-projection, and no-work-frame tests.

Everything still missing, with an actual verdict.

This ledger follows Microsoft's WPF control categories, then adds the supporting control types developers actually touch. "Missing" does not automatically mean "worth cloning." Familiar names are useful; inherited desktop baggage is not sacred.

A game UI needs navigation and history, but Cerneala should not blindly import URI navigation, desktop lifetime, journal ownership, and browser-shaped assumptions.

OpenFileDialogPrintDialogSaveFileDialog

File dialog seams already belong behind platform services. Printing only enters the plan after a real product scenario justifies a document and pagination stack.

07 / Documents and inputEvaluate

Rich text is an engine, not one control

DocumentViewerFlowDocumentPageViewerFlowDocumentReaderFlowDocumentScrollViewerRichTextBoxStickyNoteControl

These wait for multiline editing, shaping, selection, clipboard commands, pagination, accessibility, embedded elements, undo, and input method completeness.

08 / Media and inkEvaluate

Backend capability decides the shape

InkPresenterMediaElementSoundPlayerAction

InkCanvas exists as a prototype, but full WPF ink behavior does not. Media playback and presentation need backend-neutral lifetime, timing, device-loss, codec, and resource contracts.

09 / Specialist desktop UIEvaluate

Useful only when a real game or tool asks

RibbonRibbonApplicationMenuRibbonGalleryRibbonTabWebBrowserWindowsFormsHost

These do not earn roadmap priority merely because WPF shipped them. Cerneala may provide native alternatives, adapters, or nothing at all.

10 / Deliberate non-goalNot planned

Adorner, AdornerLayer, AdornerDecorator

Adorners are shit.

More technically: they introduce an opinionated second visual channel with awkward ownership, hit testing, clipping, transforms, lifetime, and composition rules. Cerneala will prefer explicit overlay roots, popup layers, template parts, diagnostics overlays, and render surfaces. No migration plan. No apology tour.

Definition of progress

A circle turns green only when the boring parts pass.

01Architecture

Backend-neutral core, explicit ownership, no duplicate subsystem, and a contract that survives more than the first demo.

02Behavior

Keyboard, pointer, focus, routed events, layout, templates, invalidation, accessibility, and failure paths all agree.

03Performance

Idle frames stay idle, hot paths allocate predictably, caches invalidate correctly, and large trees have measured budgets.

04Proof

Focused unit tests, integration tests, visual validation, a real sample, current API docs, and an honest limitations section.
